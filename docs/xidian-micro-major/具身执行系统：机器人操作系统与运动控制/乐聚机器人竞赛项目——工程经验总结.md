## 一、项目概况

| 维度 | 说明 |
|------|------|
| **平台** | 乐聚 Aelos Smart（树莓派 + ROS） |
| **任务** | 视觉闭环状态机搬运系统——识别色块 → 抱箱 → ARTag 导航 → 放箱 → 返程 |
| **代码规模** | 核心 `code.py` 443 行，435 个 `.js` 动作文件，14 个 Lua 配置文件 |
| **历史迭代** | 15 个版本存档在 `Archive/`，另有 `auto_robot/`、`manual_robot/`、`vision/`、`test/` 四个并行项目 |
| **比赛** | 中国人工智能大赛 2026——自动机器人赛项 |

## 二、技术架构

### 2.1 6 状态闭环状态机

```
起点 → 搜方块(HSV) → 抱箱 → ARTag导航(×4) → 放箱 → 返程 → 循环
```

| 步骤 | 函数 | 核心逻辑 |
|------|------|---------|
| **Step 0** | `handle_step_0()` | 搜索色块 → 确认后初始化 |
| **Step 1** | `handle_step_1()` | HSV 定位方块 → `goto_box()` 对正抓取 |
| **Step 2** | `handle_step_2()` | 抱箱转向出口 → 导航至 4 个 ARTag 标签位置 |
| **Step 3** | `handle_step_3()` | 放置箱子 → `turn_to_tag()` 精确对正 |
| **Step 4** | `handle_step_4()` | 返回起点 → 等待下一轮 |

### 2.2 技术栈

| 层 | 技术 | 强耦合点 |
|----|------|---------|
| **机器人平台** | `rospy`、`leju.base_action`、`leju.colour_port` | 乐聚专用接口，换平台即失效 |
| **视觉** | OpenCV（HSV 色块检测）+ `ar_track_alvar`（AR Tag 位姿） | 依赖胸前摄像头 ROS topic |
| **动作执行** | `base_action.action("字符串动作名")` | 动作名映射到机器人 Flash 中的 `.js` 文件 |
| **硬件** | 乐聚专用舵机、胸前 `usb_cam` 摄像头 | 完全绑定 |

### 2.3 关键算法

**HSV 色块定位** (`find_box`)：多颜色范围匹配 → 轮廓查找 → 面积阈值过滤 → 外形轮廓椭圆拟合 → 外接圆圆心坐标。

**ARTag 闭环对正** (`turn_to_tag`)：
```
获取 Tag 位姿 (x, y, θ) → 计算与目标偏移 → 
侧移补偿 x → 前进补偿 y → 旋转补偿 θ → 
循环至误差小于阈值 → 对正完成
```

> 最大的工程价值：`turn_to_tag()` 对正算法思路通用——任何带 ARTag 的机器人导航都可以复用，只需要换掉底层的 `base_action.action()` 调用。

## 三、代码特点

### 3.1 优点

| 优点 | 体现 |
|------|------|
| **状态机清晰** | 6 步函数串行，逻辑一目了然 |
| **视觉闭环** | 每步前后都有图像确认，不依赖开环计时 |
| **注释完善** | 每个函数均有中文说明，全局变量定义清楚 |
| **经验参数量化** | 注释标注了阈值来源和实测经验值 |

### 3.2 缺点

| 缺点 | 表现 | 根因 |
|------|------|------|
| **单文件 443 行** | 无模块化拆分 | 比赛节奏快，迭代优先于架构 |
| **全局变量传递状态** | `step`、`marker`、`Chest_img`、`chest_circle_x/y` | 缺乏状态管理机制 |
| **硬编码阈值** | 像素坐标 275/295/345/365、HSV 范围 | 针对特定场地标定 |
| **动作字符串调用** | `Action("GrabCube")` | 无类型安全，拼写错误运行时才暴露 |
| **无可配置参数** | 无配置文件、无命令行参数 | 每次调参需改代码 |

## 四、项目演化轨迹

从 `Archive/` 中 15 个版本可以观察到竞赛代码的典型迭代模式：

| 版本 | 特征 |
|------|------|
| `code2.py` | 早期双版本并行，`code.py`负责自动、`code2.py`负责手动 |
| `code21.py` → `code31.py` | 逐步合并功能，引入 `handle_step_N()` 结构 |
| `botec_code_v0.py` → `botec_code_nb.py` | 重构为统一状态机，`nb` 版引入线程获取图像 |
| `botec_code_merged.py` | 自动+手动融合 |
| `botec_code_lightweight.py` | 最终精简版，去除调试代码 |

> 演化规律：每个版本只改一处关键逻辑，小步迭代、快速验证——这是竞赛开发的正确节奏。

## 五、核心经验教训

### 5.1 什么是可以复用的

1. **状态机模式**：6 步视觉闭环串行——通用模式，适用于所有"感知→执行→确认"类任务
2. **`turn_to_tag()` 对正算法**：ARTag 闭环对正的核心逻辑——侧移→前进→旋转→循环
3. **HSV 标定经验**：比赛现场灯光变化巨大，每次上场前必须重新标定，代码需预留快速标定入口

### 5.2 什么是教训

1. **永远做模块化**：哪怕比赛再赶，至少把视觉/动作/状态机分三个文件
2. **动作名改枚举或常量**：`"GrabCube"` 字符串改 `Actions.GRAB_CUBE`，避免拼写错误排查半小时
3. **阈值提取为配置**：每次调像素坐标都要翻代码——一个 YAML 配置文件能省大量时间
4. **日志比 print 强**：全局加一个 `debug_level` 开关，比手动注释 `print()` 高效得多
5. **机器人操作必加紧急停止**：443 行没有一句异常处理——比赛现场机器人失控撞墙是最常见事故

### 5.3 竞赛心态

- 比赛现场没时间改架构——赛前准备决定一切
- 灯光变化是最大的不确定因素——视觉阈值必须能快速调
- 435 个 `.js` 动作文件本质是技术债务——能用 Python 控制的就不要依赖 Flash 文件

## 六、如果重来会怎么做

| 改进项 | 方案 |
|--------|------|
| **模块化拆分** | `vision.py`（HSV/AR）+ `action.py`（动作封装）+ `state_machine.py`（主逻辑） |
| **配置外部化** | YAML 配置文件：色块 HSV 范围、Tag 对正阈值、地图路径坐标 |
| **动作接口** | 枚举常量替代字符串 + 执行日志 |
| **异常处理** | 每步超时检测 + 紧急停止键（物理按钮或键盘） |
| **测试** | 视觉模块离线测试（录制的 rosbag 回放）+ 动作序列模拟模式 |
