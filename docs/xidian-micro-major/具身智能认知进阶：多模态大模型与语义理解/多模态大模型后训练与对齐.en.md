## I. Post-Training Overview

Pre-training lets the model "know"; post-training makes the model "useful", "safe", and "obedient".

| Stage | Goal | Method | Data | Cost |
|-------|------|--------|------|------|
| **Pre-training** | Learn general knowledge | Next-token prediction | TB-level web crawl | $$$$$ |
| **SFT** | Learn instruction format | Supervised learning | Tens of thousands to millions of human demonstrations | $$$$ |
| **Preference Alignment** | Align with human values | RLHF / DPO | Tens to hundreds of thousands of human preference rankings | $$$ |
| **Safety Alignment** | Refuse dangerous requests | Constitutional AI / Red Teaming | Adversarial examples + constitutional rules | $$ |

## II. SFT — Supervised Fine-Tuning

Teaches the model "HOW to answer", not "WHAT to know".

### 2.1 What SFT Can and Cannot Do

| ✅ SFT Can Do | ❌ SFT Cannot Do |
|---------------|------------------|
| Learn response format ("User: / Assistant:") | Judge which of two responses is better |
| Learn to follow instructions rather than continue text | Refuse to answer harmful questions |
| Learn multi-turn dialogue | Self-correct erroneous reasoning |
| Use specific tones (formal/friendly/professional) | Surpass the quality ceiling of training data |

> SFT determines the model's "floor" — format and style. Alignment determines the model's "ceiling" — quality and safety.

## III. RLHF — Reinforcement Learning from Human Feedback

### 3.1 Three-Step Detailed Process

```
────────── Step 1: SFT ──────────
  Pre-trained Model
  + 10K-100K high-quality (prompt, human_answer) pairs
  → SFT Model (answers following instruction format)

────────── Step 2: Train RM ──────────
  Same prompt → SFT Model generates 4-9 different responses
  → Human annotators rank these responses (A > B > C > D)
  → Train a Reward Model to predict "human preference ranking"
  → RM input: (prompt, response), output: scalar reward

────────── Step 3: PPO ──────────
  SFT Model (policy) + RM (reward signal)
  → Policy generates responses → RM scores → PPO updates policy
  → Simultaneously constrain policy not to drift too far from SFT (KL penalty)
  → Result: Aligned Model
```

### 3.2 Reward Model Training

```python
# RM training data: human preference between two responses
{ "prompt": "What is the tallest mountain in the world?",
  "chosen": "Mount Everest, altitude 8848.86 meters, located on the border between China and Nepal",
  "rejected": "Mount Everest, very tall" }

# Bradley-Terry preference model
# P(chosen > rejected) = sigmoid(r_chosen - r_rejected)
# Training objective: chosen score > rejected score

loss = -torch.log(torch.sigmoid(reward_chosen - reward_rejected))
```

### 3.3 Four Major Pain Points of RLHF

| Pain Point | Specific Manifestation | Mitigation Strategy |
|------------|-----------------------|---------------------|
| **Reward hacking** | Model finds loopholes in RM, outputs long but useless responses | KL penalty + periodic RM updates |
| **Training instability** | PPO loss oscillates, convergence difficulty | Small batch + low lr + early stopping |
| **Human annotation cost** | High-quality preference data $0.5-2/item | Synthetic data (RLAIF) |
| **Evaluation difficulty** | No single metric measures "alignment degree" | MT-Bench, AlpacaEval, Chatbot Arena |

## IV. DPO — Direct Preference Optimization

### 4.1 Core Insight

RLHF requires training a "proxy" reward model, then optimizing with PPO. DPO asks: **Can we train directly from preference data, skipping the reward model?**

The answer is yes — through reparameterization, DPO finds a closed-form solution for the optimal policy:

$$L_{DPO} = -\mathbb{E}_{(x, y_w, y_l) \sim D}\left[\log\sigma\left(\beta \log\frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log\frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right)\right]$$

### 4.2 Intuitive Understanding of DPO

| Term | Meaning | What DPO Is Doing |
|------|---------|-------------------|
| $\log\frac{\pi_\theta(y_w\|x)}{\pi_{ref}(y_w\|x)}$ | Relative improvement of the chosen response | Wants this value to be **large** (winner probability increases relative to ref) |
| $\log\frac{\pi_\theta(y_l\|x)}{\pi_{ref}(y_l\|x)}$ | Relative improvement of the rejected response | Wants this value to be **small** (loser probability should not increase) |
| $\beta$ | Controls update magnitude | Larger = more conservative; 0.1-0.5 common |

### 4.3 DPO vs RLHF Comprehensive Comparison

| Dimension | RLHF | DPO |
|-----------|------|-----|
| Needs to train reward model? | ✅ Yes | ❌ No |
| Training algorithm | PPO (reinforcement learning) | Standard cross-entropy (supervised learning) |
| Memory | Need to load 3-4 models simultaneously | 2 models (Policy + Ref) |
| Convergence speed | Slow | Fast |
| Training stability | Poor (common RL issue) | Good |
| Theoretical elegance | Empirically validated | Has closed-form solution |
| Industry adoption | ChatGPT, Claude (early) | Llama 3, Qwen 2, DeepSeek-V2 |

## V. DPO Variants

| Variant | Improvement | Applicable Scenario |
|---------|-------------|---------------------|
| **IPO** | Adds regularization to prevent overfitting | Noisy data |
| **KTO** | Does not need pairwise preferences — only single feedback (good/bad) | Sparse feedback signals |
| **ORPO** | Merges SFT and alignment into a single stage | Simplifies training pipeline |
| **SimPO** | Uses sequence length as implicit regularization | Avoids length bias |
| **Iterative DPO** | Each round uses the updated model to generate new preference pairs | Continuous self-improvement |

## VI. Constitutional AI — Principle-Based Alignment

Rather than relying on human-annotated preferences, it **presets constitutional principles** to automatically generate training data.

```
Constitutional Principles:
1. Choose the response that is least harmful to humans
2. Choose the response that does not contain racial discrimination
3. Choose the response that does not encourage illegal behavior
4. Choose the most honest response — if uncertain, choose "I don't know"
```

### Constitutional AI Process

```
Phase 1: Supervised Stage
  Model generates harmful response
  → Constitution critiques the response and points out issues
  → Model revises the response based on the critique
  → Revised response used as training data → SFT

Phase 2: RL Stage
  Model generates response
  → Constitution-based AI judges which is better (replaces humans)
  → Generates preference data → RLHF or DPO training
```

> Claude is the most successful practice of Constitutional AI — its safety leads across multiple benchmarks.

## VII. Evaluation System

| Benchmark | What It Tests | Method |
|-----------|---------------|--------|
| **AlpacaEval** | Response quality vs GPT-4 | LLM-as-Judge automatic scoring |
| **MT-Bench** | Multi-turn dialogue + reasoning + writing | GPT-4 scoring |
| **Chatbot Arena** | Real human voting | A/B test |
| **HellaSwag** | Commonsense reasoning | Multiple choice accuracy |
| **TruthfulQA** | Truthfulness — no lying | Multiple choice accuracy |
| **SafetyBench** | Safety | Rejection rate + accuracy |

## VIII. Special Challenges of Multimodal Alignment

| Challenge | Specific Problem | Current Exploration |
|-----------|-----------------|---------------------|
| **Visual bias** | "Nurse = female" "CEO = male" | Diverse training data |
| **Privacy leakage** | Segmentation models may output faces from training images | Differential privacy training |
| **Image jailbreaking** | Embedding hidden prompts in images to attack the model | Input-layer defenses |
| **Multilingual alignment** | English RLHF does not guarantee Chinese also safe | Multilingual preference data |
| **Multimodal hallucination** | Describing objects not present in the image | RLHF on VQA pairs |
