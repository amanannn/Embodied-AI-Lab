## 一、什么是 AI Agent

AI Agent 是能够**自主感知环境、制定计划、调用工具、执行行动**的智能体。

| 能力 | 说明 | 示例 |
|------|------|------|
| **感知** | 理解环境状态和用户意图 | 读取对话上下文 |
| **规划** | 分解任务为可执行步骤 | "预订餐厅"→搜索→筛选→预订 |
| **工具使用** | 调用外部 API/函数 | `search_restaurants()`、`send_email()` |
| **记忆** | 短期（对话历史）+ 长期（知识库） | 记住用户偏好 |
| **反思** | 评估自己的输出并修正 | "回答不对，重新查" |

> Agent ≠ Chatbot。Chatbot 被动回答；Agent 主动执行——设定目标、拆解步骤、调用工具、验证结果。

## 二、Agent 发展三阶段

| 阶段 | 时期 | 代表 | 特征 |
|------|------|------|------|
| **早期探索** | 2018-2022 | ReAct, SayCan, Toolformer | 学术原型——证明 LLM 能调用工具 |
| **工程化** | 2023 上半年 | AutoGPT, BabyAGI, LangChain Agent | 开源框架爆发——人人都能搭 Agent |
| **产品化** | 2023 下半年至今 | GPT-4 Function Calling, Claude Computer Use, OpenAI Swarm | 平台级产品——Agent 成为 LLM 标配能力 |

## 三、经典 Agent 框架

### 3.1 ReAct——推理 + 行动交替

ReAct（Reasoning + Acting）是 Agent 架构的奠基之作：

```
Thought: 我需要查找今天的天气
Action: search_weather("北京", "2024-07-16")
Observation: 北京今天晴，25-35°C
Thought: 用户还想知道适合穿什么
Action: recommend_clothing("25-35°C", "晴")
Observation: 建议短袖+防晒
Thought: 我已获得所有信息，可以回答了
Answer: 北京今天晴，25-35°C，建议穿短袖...
```

| 组件 | 作用 |
|------|------|
| **Thought** | 思考下一步该做什么 |
| **Action** | 执行具体操作（搜索/计算/API 调用） |
| **Observation** | 接收操作结果 |
| **（循环）** | 直到获得足够信息回答问题 |

### 3.2 AutoGPT——自主任务执行

```
用户目标: "帮我调研 AI Agent 赛道，输出一份报告"
    ↓
Task: 搜索"AI Agent 2024 年发展"
    → 记忆: 保存搜索结果
    → 分析: 提取关键信息
    → 子任务: 搜索"Agent 细分赛道"
    → 子任务: 搜索"Agent 市场规模"
    → 整合: 融合信息生成报告
    → 反思: 检查报告完整性
    → 输出: 最终报告
```

| 特点 | 说明 |
|------|------|
| **长程自主** | 能运行数十分钟，执行几十步 |
| **记忆系统** | 短期（Redis/内存）+ 长期（向量数据库） |
| **问题** | 容易发散——任务拆解后越来越偏离目标 |

### 3.3 GPT-4 Function Calling——工业标准

| 优势 | 说明 |
|------|------|
| **结构化输出** | 模型不"猜"答案，而是明确说要调哪个函数 |
| **可靠性高** | 参数由开发者控制，不会模型幻觉生成 |
| **可组合** | 多个 function call 串行/并行 |

### 3.4 Multi-Agent——多个 Agent 协作

```
用户任务 → Orchestrator Agent
    ├── Researcher Agent: 搜索信息
    ├── Analyst Agent: 分析数据
    ├── Writer Agent: 撰写输出
    └── Reviewer Agent: 检查质量
```

| 代表框架 | 特点 |
|---------|------|
| **AutoGen** (Microsoft) | 多 Agent 对话，支持代码生成和执行 |
| **CrewAI** | 角色定义清晰，适合工作流类任务 |
| **OpenAI Swarm** | 轻量多 Agent 编排，实验性质 |
| **LangGraph** | 有状态图——Agent → 节点，交互 → 边 |

## 四、Agent 工程框架

| 框架 | 核心思路 | 适用场景 |
|------|---------|---------|
| **LangChain** | 链式调用——Prompt → LLM → Tool → LLM | 快速原型 |
| **LangGraph** | 图状态机——节点是 Agent，边是交互 | 复杂多步 Agent |
| **LlamaIndex** | 数据索引 + 查询引擎 | 知识密集型 Agent |
| **Semantic Kernel** | 插件式——技能/记忆/规划三大组件 | 企业级 Agent |
| **Dify / Coze** | 拖拽式低代码 | 非开发者用 Agent |

## 五、Agent 的核心挑战

| 挑战 | 说明 | 当前方案 |
|------|------|---------|
| **幻觉** | 编造不存在的函数调用结果 | Function Calling 结构化约束 |
| **发散** | 多步任务偏离原始目标 | 目标状态机 + 定期检查 |
| **速度** | 10+ 步推理太慢 | 流式输出 + 并行执行 |
| **成本** | 每步都调大模型 | 小模型做简单步骤 |
| **安全** | Agent 可能删除文件/发送错误邮件 | 权限管理 + 人类确认 |
