## I. Fine-Tuning: From Generalist to Specialist

| Stage | Data Volume | Compute Volume | Goal | Analogy |
|-------|-------------|----------------|------|---------|
| **Pre-training** | TB-level unlabeled | Thousands of GPU·months | General knowledge | Elementary school to university |
| **Fine-tuning** | KB~GB-level labeled | Single to dozens of GPUs | Specific task | On-the-job training |

## II. Full Fine-Tuning

Updates **all parameters** of the model — highest ceiling, greatest cost.

| Model Size | Parameter Memory (FP16) | Training Memory (incl. optimizer + gradient) | Minimum GPU |
|-----------|------------------------|----------------------------------------------|-------------|
| 1B | ~2 GB | ~10 GB | RTX 3090 (24G) |
| 7B | ~14 GB | ~60 GB | 2×A6000 (48G) |
| 13B | ~26 GB | ~110 GB | 4×A100 (80G) |
| 70B | ~140 GB | ~560 GB | 8×A100 (80G) |

> Full fine-tuning of a 7B model requires approximately 60GB of VRAM — far exceeding a single consumer-grade GPU. This is the reason PEFT exists.

## III. LoRA — Low-Rank Adaptation

### 3.1 Core Mathematics

The update to pre-trained weights $W_0 \in \mathbb{R}^{d \times k}$ is constrained to a low-rank decomposition:

$$W = W_0 + \Delta W = W_0 + BA$$

where $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, $r \ll \min(d, k)$.

The forward pass becomes:

$$h = W_0 x + BAx = W_0 x + \frac{\alpha}{r} \cdot BAx$$

| Parameter | Meaning | Typical Value | Tuning Advice |
|-----------|---------|---------------|---------------|
| **r (rank)** | Low-rank dimension — determines trainable parameters | 8 (economical) / 64 (precision-first) | Simple tasks: r=8 is enough; complex multi-task: r=64 |
| **α (alpha)** | Scaling factor | 2~4× r | α=2r is a common starting point |
| **target_modules** | Which layers to apply LoRA | `q_proj,v_proj` (minimum) → all attention layers | More layers = higher precision but more parameters |
| **lora_dropout** | Regularization | 0.05~0.1 | Increase when data is scarce |

### 3.2 Why Does LoRA Work?

Although the pre-trained weight matrix $W$ has large dimensions, the **weight update matrix $\Delta W$ often has a low "intrinsic rank"** — fine-tuning does not need to search the full space; it only needs to adjust along a few important directions. Aghajanyan et al. (2021) demonstrated this.

### 3.3 Complete Fine-Tuning Script

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from peft import LoraConfig, get_peft_model, TaskType
from datasets import load_dataset
import torch

# 1. Load model
model_name = "Qwen/Qwen2-7B-Instruct"
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto",
    trust_remote_code=True,
)
tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True)
tokenizer.pad_token = tokenizer.eos_token

# 2. Configure LoRA
lora_config = LoraConfig(
    r=16,                            # rank
    lora_alpha=32,                   # scaling
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],  # full attention + FFN
    lora_dropout=0.05,
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 41,943,040 || all params: 7,615,610,880 || trainable%: 0.55%

# 3. Prepare data
def format_instruction(example):
    return {"text": f"<|user|>\n{example['instruction']}\n<|assistant|>\n{example['output']}"}

dataset = load_dataset("yahma/alpaca-cleaned")
dataset = dataset.map(format_instruction)

# 4. Train
trainer = Trainer(
    model=model,
    args=TrainingArguments(
        output_dir="./qwen-lora-alpaca",
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,    # effective batch_size=16
        learning_rate=2e-4,
        num_train_epochs=3,
        logging_steps=10,
        save_strategy="epoch",
        bf16=True,                        # A100/H100 use bf16
        gradient_checkpointing=True,      # trade speed for memory
    ),
    train_dataset=dataset["train"],
)
trainer.train()

# 5. Save — only the adapter (a few MB)
model.save_pretrained("./qwen-lora-alpaca-adapter")
```

## IV. QLoRA — Quantized Fine-Tuning

### 4.1 Four-Fold Optimization Techniques

| Technique | Effect | Memory Savings |
|-----------|--------|----------------|
| **4-bit NormalFloat** | Information-theoretically optimal 4-bit quantization | ~4× |
| **Double Quantization** | Quantizing the quantization constants again | ~0.4 bit/parameter |
| **Paged Optimizers** | Automatically evict optimizer states to CPU when OOM | 30% peak reduction |
| **LoRA** | Only train low-rank adapters | ~100× (parameter count) |

### 4.2 QLoRA Fine-Tuning Script

```python
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",           # NormalFloat4
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,      # Double quantization
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-70b-chat-hf",
    quantization_config=bnb_config,
    device_map="auto",
    trust_remote_code=True,
)
# 70B model → ~35 GB → single A6000 (48G) is enough!
```

### 4.3 Full Memory Comparison

| Method | 7B | 13B | 34B | 70B | 405B |
|--------|-----|-----|-----|-----|------|
| Full fine-tuning | ~60 GB | ~110 GB | Not feasible | Not feasible | Not feasible |
| LoRA (FP16) | ~16 GB | ~30 GB | ~80 GB | ~160 GB | Not feasible |
| QLoRA (4bit) | ~6 GB | ~10 GB | ~18 GB | ~35 GB | ~80 GB |

## V. LoRA Variants

| Variant | Improvement | Use Case |
|---------|-------------|----------|
| **LoRA+** | Different learning rates for $B$ and $A$ | General |
| **AdaLoRA** | Adaptive rank allocation per layer | Pursuing extreme parameter efficiency |
| **DoRA** | Decomposes weights into magnitude + direction, fine-tunes only direction | Approaches full fine-tuning quality |
| **QLoRA** | LoRA + 4-bit quantization | Fine-tuning large models on consumer GPUs |
| **LoRA-FA** | Freeze $A$, train only $B$ | Halves parameter count further |

### 6.1 Data Format

```json
[
  {
    "instruction": "Add comments to the following code",
    "input": "def fib(n):\n    if n<=1: return n\n    return fib(n-1)+fib(n-2)",
    "output": "def fib(n):\n    \"\"\"Compute the nth Fibonacci number\"\"\"\n    if n <= 1:\n        return n\n    return fib(n - 1) + fib(n - 2)"
  }
]
```

### 6.3 Vision-Language Fine-Tuning

| Strategy | Operation | When to Use |
|----------|-----------|-------------|
| **LoRA LLM only** | Freeze ViT + Projector → LoRA fine-tune LLM | Preserve vision capabilities, improve dialogue |
| **LoRA LLM + Unfreeze Projector** | Full Projector + LoRA LLM | Adapt to new vision encoder |
| **LoRA LLM + Unfreeze ViT last layers** | Slightly adjust visual features | Visual domain shift (medical/remote sensing) |
| **Full** | Everything trainable | Lots of data + plenty of GPU |

## VII. Fine-Tuning Practical Checklist

- [ ] Confirm the base model matches the task — do not use a pure text model for multimodal fine-tuning
- [ ] Align data format — instruction/input/output field names should match the model's pre-training
- [ ] First do LoRA quick validation (r=8) — confirm loss decreases before increasing rank
- [ ] Monitor train loss + eval loss — loss not decreasing = problem with learning rate / data
- [ ] When using QLoRA — unfrozen layers cannot be 4-bit (need to maintain precision)
- [ ] Save the adapter, not the full model — a few MB vs. tens of GB
- [ ] After fine-tuning, do qualitative testing — manually test with 5-10 real cases
- [ ] Record key parameters — r, α, lr, batch_size, epoch — for reproducibility
